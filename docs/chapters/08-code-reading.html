コードを読む力 —— 2026年に人間に残された必須スキル

「書く」はAIに任せていい。でも「読む」は人間の責任。——その理由と、最低限の習得方法。

1. Anthropicの研究が証明した事実

2026年1月、Anthropicが52人のジュニアエンジニアで実験を行いました。

実験内容

Pythonの新しいライブラリ（Trio）を学んでもらい、AI補助ありグループと手書きグループで比較。タスク後に理解度テスト（コード読解・デバッグ・概念理解）を実施。

結果

AIの使い方 理解度テストのスコア
AIに完全に丸投げ（コード生成のみ） 24〜39%
AIに概念的質問 / 説明を求める 65〜86%
手書き（AI未使用） 67%

完全にAIに丸投げした人は、何も理解していなかった。

Anthropic研究チームの結論

「AIによる生産性向上は、能力への近道ではない」

「特に安全上重要な領域では、AI生成コードを検証するスキルが求められる。しかしAIへの依存がそのスキル形成を阻害する可能性がある」

出典: Anthropic Research, "How AI assistance impacts the formation of coding skills" (2026年1月)

2. 「書けない」と「読めない」は別物

書く（Writing） 読む（Reading）
AIに任せる？ ✅ 任せるべき ❌ 任せきれない
習得難易度 高い（月〜年） 低い（週〜月）
必要量 ゼロでOK 最低限は必須
比喩 大工が壁を建てる技術 建築家が図面を読む技術

「コードが書けない」は選択です。AIが人間より速いから任せる。——これは2026年の合理判断です。

「コードが読めない」は弱点です。図面が読めない建築家と同じ。

3. 「読めない」が致命的な3つの理由

① チーム開発では「読む」ことが仕事の半分

場面 必要な能力
PRレビュー 「この変更は何をしているか」を判断する
インシデント対応 エラーログ・スタックトレースを読む
設計議論 「この実装のトレードオフは？」に答える
ペアプロ・モブプロ 画面上のコードをリアルタイムで追う

会議中に「ちょっとAIに聞いてきます」とは言えません。

② 「品質判定」が売りなのに、読めないと判定できない

私のスタンスは「要件定義・品質判定・運用設計を指揮する」です。

でも品質判定の対象がコードです。コードが読めない状態で品質判定をするのは、図面が読めない建築家が検査印章を押すのと同じ。

③ AIが書いたコードの「最終責任」は人間

Anthropicの研究が指摘：

「AI生成コードをデプロイ前に理解し検証する能力。このスキルは、AIがコードを書く時代にこそ最も重要になる」

4. 必要な「最低限の読む力」

完全なプログラマーになる必要はありません。以下の3レベルができれば実務で困らないラインです。

Level 1: エラーメッセージが読める（最重要・1〜2週間）

Traceback (most recent call last):
  File "app.py", line 42, in process_order
    total = price * quantity
TypeError: unsupported operand type(s) for *: 'str' and 'int'

読み方:

  • File "app.py", line 42 → どこで起きたか
  • TypeError → エラーの種類（型が違う）
  • unsupported operand type(s) → 掛け算できない組み合わせ
  • 'str' and 'int' → 文字列と数値

→ 「42行目で、文字列と数値を掛け算しようとしてエラー」

これが読めれば、バグ報告の精度が劇的に上がります。

Level 2: PRのdiffが読める（2〜4週間）

- def calculate_total(price):
+ def calculate_total(price, tax_rate=0.1):
+     return price * (1 + tax_rate)
-     return price

読み方:

  • -（赤）は消えた行、+（緑）は追加された行
  • 関数にtax_rate引数が追加された
  • デフォルト値は0.1（10%）
  • 戻り値が「価格×（1+税率）」に変わった

→ 「消費税率の引数が追加されて、合計に税が含まれるようになった」

これが読めれば、コードレビューに参加できます。

Level 3: 全体の流れが追える（1〜2ヶ月）

@app.route("/api/orders", methods=["POST"])   # ① URLとHTTPメソッドの定義
def create_order():
    data = request.json                        # ② 入力を受け取る
    order = Order(**data)                      # ③ DB用オブジェクトに変換
    db.session.add(order)                      # ④ DBに保存
    db.session.commit()                        # ⑤ 保存を確定
    return jsonify(order.to_dict()), 201       # ⑥ レスポンス返す

→ 「APIがリクエストを受け取ってDBに保存して返す流れ」

これが読めれば、設計議論に参加できます。

必要ないもの

  • アルゴリズムの理解
  • データ構造の暗記
  • 複雑な構文の記述
  • ホワイトボードコーディング
  • ゼロからのコード作成

5. 正しい学習方法（Anthropic推奨パターン）

Anthropicの研究が示したスコア65〜86%のパターンをそのまま適用します。

❌ やらないこと（スコア24〜39%）

  • AIに「このコードを書いて」とだけ頼む
  • エラーが出たらAIに丸投げして直してもらう
  • コードをコピペして終わり

✅ やること（スコア65〜86%）

  • 「このコードが何をしているか説明して」
  • 「このエラーの原因は何？」
  • 「この変更の意図は何？」
  • 「なぜこの実装になっているの？」

AIに「説明させる」ことで、自分の理解が深まる。 これがAnthropic推奨の正しい使い方です。

面接で聞かれたら

Q: 「コードは読めるのですか？」

実装はAIに任せていますが、コードの読解には力を入れています。PRレビューで変更内容を理解し、エラーメッセージから原因を特定し、全体のアーキテクチャの流れを追うことは日常的にやっています。

Anthropicの2026年の研究で、AIに丸投げする人は理解度が24〜39%に落ちる一方、AIに「説明を求める」使い方をすれば65〜86%の理解度を維持できることが分かっています。私は後者の使い方を意識しています。

用語の平易な解説（専門用語を読み解く）

🐛 Traceback（トレースバック）・スタックトレース

一言で: 「エラーが起きた時の、通過経路の記録」。

  • Traceback: Pythonでエラーが起きた時に表示される、処理の追跡記録。「どこ→どこ→どこで止まった」が順に表示される
  • スタックトレース: 同じ意味の一般用語。関数の呼び出し履歴を遡って表示する
  • 読み方のコツ:一番下の行に「本当の原因（エラーの種類）」が書いてある

🔢 TypeError（タイプエラー）・型（かた）

一言で: 「データの種類（型）が合わず起きるエラー」。

  • 型（かた）: データの種類。「文字列（str）」「数値（int）」等。Excelでいう「文字か数値か」の区別
  • TypeError: 「文字列と数値を掛け算しようとした」等、型の組み合わせがおかしい時に起きるエラー
  • 例：「"100" × 3」は「100100100」（文字の反復）になるか「300」（数値の掛け算）か曖昧。Pythonでは型を明確に区別する

🔀 diff（ディフ）

一言で: 「2つのバージョンの差分（変更内容）」。

  • diff: 変更前と変更後を比べて「どこが変わったか」を表示したもの
  • 表示ルール: -（赤）＝消えた行、+（緑）＝追加された行
  • PRレビューの基本は、この diff を読んで「何をどう変えたか」を把握すること

👁️ PRレビュー・PR（プルリクエスト）

一言で: 「提案されたコード変更を、他の人が確認する作業」。

  • PR（Pull Request・プルリクエスト）: 「この変更を本体に取り込んでください」という提案。「マージ（統合）の依頼」
  • PRレビュー: 提案された変更を、バグがないか・読みやすいか等の観点で確認すること。本章では「レビューに参加するには diff が読めが必要」

👥 ペアプロ・モブプロ

一言で: 「2人〜複数人で1つのコードを見ながら進める開発スタイル」。

  • ペアプロ（ペアプログラミング）: 2人で1台のパソコンを囲んで作業する手法
  • モブプロ（モブプログラミング）: 3人以上で1台のパソコンを囲む手法
  • いずれも「画面上のコードをリアルタイムで追う読む力」が必要

📋 ホワイトボードコーディング

一言で: 「面接等で、ホワイトボードに手書きでコードを書かせる試験」。

  • 昔の技術面接でよく行われた「その場でコードを書いてください」という課題
  • AI駆動開発の時代には不要になりつつある。本章でも「必要ないもの」に挙げている
